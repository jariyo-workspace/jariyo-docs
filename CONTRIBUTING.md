# 기여 가이드

자리요 문서 저장소는 `Git Flow` 기반으로 운영한다. 모든 변경은 브랜치, 커밋, PR 규칙을 따라야 한다.

## 브랜치 규칙

* 모든 작업은 먼저 이슈를 생성한 뒤 시작한다.
* `main`에는 직접 커밋하지 않는다.
* 모든 작업은 목적에 맞는 브랜치에서 진행한다.
* 브랜치 이름은 작업 성격이 드러나야 한다.

예시:
* `feature/reservation-status-policy`
* `fix/api-response-typo`
* `docs/readme-link-update`

## 커밋 규칙

* 모든 커밋 메시지에는 반드시 prefix를 붙인다.
* 한 커밋에는 하나의 목적만 담는다.
* 의미 없는 메시지는 사용하지 않는다.

### Prefix 목록

* `feat:` 새로운 기능, 정책, 섹션, 문서 내용 추가
* `fix:` 오타, 오류, 잘못된 설명, 잘못된 예시 수정
* `docs:` 문서 표현 개선, 링크 수정, 구조 정리
* `refactor:` 의미 변화 없이 문서 구조 또는 구성 정리
* `chore:` 분류하기 애매한 기타 작업

예시:
* `docs: API 명세 예약 상태 설명 보완`
* `fix: 화면 기획서 오타 수정`
* `feat: 대기 순서 정책 문서 추가`

## PR 규칙

* 모든 변경은 PR로만 반영한다.
* 작업 순서는 반드시 `이슈 생성 -> 브랜치 작업 -> PR 생성 -> 머지 후 이슈 종료`를 따른다.
* PR 제목은 반드시 `[PREFIX] 제목` 형식으로 작성한다.
* PR 하나에는 하나의 주제만 담는다.
* 관련 문서가 함께 수정되어야 하면 같은 PR에 포함한다.
* PR 본문에는 반드시 연결된 이슈를 명시한다.

예시:
* `[DOCS] 예약 상태 정의 정리`
* `[FIX] API 응답 예시 오타 수정`

허용 PR prefix:
* `[FEAT]`
* `[FIX]`
* `[DOCS]`
* `[REFACTOR]`
* `[CHORE]`

이슈 연결 예시:
* `Closes #12`
* `Resolves #34`

## 머지 규칙

* 리뷰 없이 머지하지 않는다.
* 관련 없는 커밋은 정리한 뒤 머지한다.
* PR 범위가 크면 주제별로 나누어 올린다.

## 금지 사항

* 이슈 없이 바로 작업 시작
* `main` 직접 커밋
* prefix 없는 커밋
* 여러 주제를 하나의 PR에 혼합
* `update`, `edit`, `fix stuff` 같은 모호한 메시지 사용
