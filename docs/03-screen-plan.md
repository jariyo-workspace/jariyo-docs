# 자리요 화면 기획서

## 1. 문서 목적

본 문서는 자리요의 고객용 화면과 매장 운영자용 화면을 설계하기 위한 기준을 정의한다.

자리요는 예약, 예약 대기, 현장 대기를 하나의 서비스 안에서 제공하지만, 사용자가 한 화면에서 지나치게 많은 정보를 접하지 않도록 역할과 상황에 따라 화면을 구분한다.

화면 설계의 핵심 목표는 다음과 같다.

* 고객이 예약 가능한 시간을 빠르게 찾을 수 있어야 한다.
* 예약이 마감된 경우 자연스럽게 대기 신청으로 이어져야 한다.
* 고객이 현재 자신의 예약·대기 상태를 한눈에 이해할 수 있어야 한다.
* 매장 직원이 예약 고객과 현장 대기 고객을 한 화면에서 관리할 수 있어야 한다.
* 중요한 상태 변경은 실수를 줄이도록 확인 절차를 제공해야 한다.
* 처리 실패나 충돌 상황을 사용자에게 이해하기 쉬운 언어로 안내해야 한다.

# 2. 서비스 화면 구성

자리요는 크게 두 개의 서비스 영역으로 구성한다.

## 고객용 서비스

고객이 매장을 조회하고 예약하거나, 예약 대기와 현장 대기를 이용하는 화면이다.

주요 영역은 다음과 같다.

* 매장 탐색
* 서비스 및 직원 선택
* 예약 가능 시간 조회
* 예약 신청
* 예약 대기 신청
* 현장 대기 등록
* 예약 및 대기 현황 확인
* 알림 확인
* 마이페이지

## 매장 운영자용 서비스

매장 관리자와 직원이 매장 일정, 예약, 대기 고객을 관리하는 화면이다.

주요 영역은 다음과 같다.

* 오늘의 운영 현황
* 예약 일정 관리
* 현장 대기열 관리
* 직원 및 서비스 관리
* 영업시간과 휴무 관리
* 예약 대기 및 빈자리 관리
* 고객 상태 처리
* 운영 통계
* 매장 설정

# 3. 공통 디자인 원칙

## 상태를 가장 먼저 보여준다

자리요에서는 사용자의 현재 상태가 가장 중요한 정보다.

예를 들어 고객이 예약 상세 화면에 들어왔을 때 날짜나 매장 소개보다 먼저 다음 상태가 보여야 한다.

* 예약 확정
* 체크인 가능
* 서비스 진행 중
* 예약 취소
* 빈자리 제안 도착
* 현장 대기 중
* 고객 호출 중

상태는 색상만으로 구분하지 않고 아이콘과 문구를 함께 사용한다.

## 사용자에게 내부 시스템 용어를 노출하지 않는다

다음과 같은 내부 용어는 화면에 직접 노출하지 않는다.

* HELD
* CONFIRMED
* EXPIRED
* SLOT OFFER
* WAITLIST ENTRY
* PROCESSING

대신 다음과 같이 표현한다.

* 예약 확인 중
* 예약 확정
* 제안 시간이 만료되었어요
* 빈자리가 생겼어요
* 예약 대기 중
* 요청을 처리하고 있어요

## 중요한 행동은 한 화면에 하나만 강조한다

예약 상세 화면에서 상황에 따라 가장 중요한 행동 하나를 강조한다.

예:

* 예약 확정 전: `예약 확정하기`
* 방문 직전: `체크인하기`
* 빈자리 제안 도착: `이 시간 예약하기`
* 현장 호출 중: `입장 확인하기`

부가 행동은 보조 버튼이나 더보기 메뉴에 배치한다.

## 시간 정보는 절대적인 날짜와 함께 표시한다

`내일 오후 2시`만 표시하지 않고 다음처럼 함께 보여준다.

```text
7월 18일 토요일 오후 2:00
```

상대 시간은 보조적으로 사용한다.

```text
내일 · 오후 2:00
```

## 실패 상황에서도 현재 상태를 잃지 않게 한다

네트워크 오류나 처리 실패가 발생했을 때 단순히 “오류가 발생했습니다”라고 표시하지 않는다.

예:

```text
예약 요청 결과를 확인하지 못했어요.
예약 내역에서 처리 결과를 다시 확인해 주세요.
```

버튼:

```text
예약 내역 확인
다시 시도
```

# 6. 고객 화면 상세 기획

## C-01. 시작 및 로그인

### 목적

사용자가 자리요에 로그인하거나 제한된 기능을 비회원으로 이용할 수 있도록 한다.

### 주요 구성

* 자리요 로고
* 서비스 소개 문구
* 로그인 버튼
* 회원가입 버튼
* 간편 로그인 영역
* 비회원 현장 대기 조회
* 이용약관 및 개인정보 처리방침

### 대표 문구

```text
예약도 대기도 한자리에서
```

### 주요 행동

* 로그인
* 회원가입
* 비회원 대기 조회

### 예외 상황

로그인 세션이 만료된 경우 사용자가 진행하던 예약 정보를 임시 보관하고 로그인 후 이어서 진행한다.

---

## C-02. 고객 홈

### 목적

사용자가 현재 가장 중요한 예약과 대기 상태를 빠르게 확인하고 다음 행동으로 이동하게 한다.

### 화면 구성

#### 상단 영역

* 사용자 인사
* 알림 아이콘
* 매장 검색 버튼

#### 현재 이용 상태 카드

우선순위에 따라 하나의 카드를 가장 크게 노출한다.

1. 현재 호출 중인 현장 대기
2. 수락 기한이 있는 빈자리 제안
3. 오늘 예정된 예약
4. 현재 현장 대기
5. 예약 대기
6. 다가오는 예약

#### 빠른 메뉴

* 예약하기
* 현장 대기
* 내 예약
* 최근 본 매장

#### 최근 이용 매장

* 매장명
* 최근 이용 서비스
* 다시 예약 버튼

### 상태별 카드 예시

```text
빈자리가 생겼어요
오늘 오후 3:30 · 커트
02:41 안에 수락해 주세요
```

버튼:

```text
예약 확인
```

---

## C-03. 매장 검색

### 목적

사용자가 매장명, 업종 또는 지역을 기준으로 매장을 찾도록 한다.

### 주요 구성

* 검색 입력창
* 최근 검색어
* 최근 본 매장
* 업종 필터
* 영업 중 필터
* 오늘 예약 가능 필터
* 검색 결과 목록

### 매장 카드 정보

* 매장명
* 업종
* 주소 또는 거리
* 오늘 영업시간
* 가장 빠른 예약 가능 시간
* 현재 현장 대기 가능 여부

### 빈 상태

```text
검색 결과가 없어요.
다른 매장명이나 지역으로 검색해 보세요.
```

## C-04. 매장 상세

### 목적

매장 정보를 확인하고 예약 또는 현장 대기를 시작하도록 한다.

### 상단 정보

* 매장명
* 대표 이미지
* 업종
* 영업 상태
* 주소
* 전화번호
* 매장 소개

### 주요 기능 영역

* 예약하기
* 현장 대기 등록
* 서비스 목록
* 직원 목록
* 영업시간
* 예약 및 취소 정책
* 매장 공지

### 고정 하단 버튼

상황에 따라 다음 중 하나 또는 두 개를 제공한다.

```text
예약하기
현장 대기 등록
```

현장 대기를 운영하지 않는 매장은 해당 버튼을 표시하지 않는다.

## C-05. 서비스 선택

### 목적

고객이 이용할 서비스와 예상 소요 시간을 선택하도록 한다.

### 서비스 카드 정보

* 서비스명
* 설명
* 예상 소요 시간
* 예약 가능 여부
* 담당 가능한 직원 수

### 주요 행동

* 서비스 선택
* 다음 단계 이동

### 유의사항

한 번에 하나의 서비스만 선택하도록 단순화한다.

---

## C-06. 직원 선택

### 목적

고객이 희망 직원을 선택하거나 직원 무관 예약을 선택하도록 한다.

### 옵션

* 가능한 직원 누구나
* 특정 직원 선택

### 직원 카드 정보

* 직원 이름
* 프로필 이미지
* 담당 가능한 서비스
* 가장 빠른 예약 가능 시간
* 휴무 또는 예약 불가 상태

### 직원 선택 정책

특정 직원만 가능한 경우 직원 선택을 필수로 한다.

선택하지 않은 경우 시스템이 가능한 직원 중 한 명을 배정할 수 있음을 안내한다.

---

## C-07. 날짜 및 시간 선택

### 목적

고객이 예약 가능한 날짜와 시간을 선택하도록 한다.

### 화면 구성

* 선택한 서비스 요약
* 선택한 직원 요약
* 날짜 선택 캘린더
* 오전·오후 시간 목록
* 가장 빠른 시간 표시
* 마감된 시간 표시
* 예약 대기 신청 진입

### 시간 버튼 상태

* 예약 가능
* 선택됨
* 마감
* 현재 선택 불가
* 대기 신청 가능

### 예약 가능한 시간이 없는 경우

```text
선택한 날짜에 예약 가능한 시간이 없어요.
원하는 시간대를 등록하면 빈자리가 생겼을 때 알려드릴게요.
```

버튼:

```text
예약 대기 신청
다른 날짜 보기
```

### 동시 예약 충돌 발생 시

사용자가 시간을 선택했지만 다른 고객이 먼저 예약한 경우:

```text
방금 다른 고객이 이 시간을 예약했어요.
다른 가능한 시간을 선택해 주세요.
```

가능 시간 목록을 새로 갱신한다.

## C-08. 예약 확인

### 목적

예약 확정 전에 모든 정보를 검토하도록 한다.

### 정보 영역

* 매장명
* 서비스명
* 담당 직원
* 예약 날짜
* 시작 시간
* 예상 종료 시간
* 고객 정보
* 요청사항
* 취소 정책

### 동의 영역

* 예약 및 취소 정책 확인
* 개인정보 제공 동의

### 주요 버튼

```text
예약 확정하기
```

### 중복 제출 방지

버튼을 누른 뒤 처리 중에는 버튼을 비활성화한다.

```text
예약을 확인하고 있어요
```

새로고침하거나 요청이 반복돼도 사용자에게 여러 예약이 생성된 것처럼 보이지 않아야 한다.

## C-09. 예약 완료

### 목적

예약 결과를 명확히 보여주고 다음 행동을 안내한다.

### 주요 정보

* 예약 완료 아이콘
* 예약 상태
* 매장명
* 서비스
* 날짜와 시간
* 직원
* 예약 번호

### 주요 버튼

* 예약 상세 보기
* 캘린더에 추가
* 홈으로

### 안내 문구

```text
예약 시간이 가까워지면 다시 알려드릴게요.
```

## C-10. 예약 상세

### 목적

예약 상태와 방문에 필요한 정보를 확인하고 예약을 관리한다.

### 상단 상태 영역

상태별로 다음 정보를 표시한다.

#### 예약 확정

```text
예약이 확정되었어요
7월 18일 토요일 오후 2:00
```

#### 체크인 가능

```text
이제 체크인할 수 있어요
```

#### 이용 중

```text
서비스가 진행 중이에요
```

#### 취소됨

```text
취소된 예약이에요
```

### 상세 정보

* 매장
* 서비스
* 직원
* 예약 시간
* 예상 소요 시간
* 고객 요청사항
* 취소 가능 시간
* 상태 변경 이력 요약

### 주요 행동

상태에 따라 버튼을 다르게 제공한다.

* 체크인하기
* 예약 취소
* 매장에 전화
* 다시 예약
* 길찾기

### 예약 취소

취소 전 확인 모달:

```text
예약을 취소할까요?

취소한 시간은 다른 고객에게 제공될 수 있으며,
취소 후에는 되돌릴 수 없어요.
```

버튼:

```text
예약 유지
예약 취소
```

## C-11. 예약 대기 신청

### 목적

예약이 마감된 경우 고객이 원하는 조건을 등록하도록 한다.

### 입력 항목

* 매장
* 서비스
* 희망 날짜
* 가능한 시작 시간
* 가능한 종료 시간
* 선호 직원
* 다른 직원도 가능한지
* 대기 만료 시점

### 조건 요약

화면 하단에 자연어 형태로 보여준다.

```text
7월 18일 오후 1시부터 5시 사이,
민지 디자이너를 우선으로 기다릴게요.
다른 직원도 가능합니다.
```

### 주요 버튼

```text
예약 대기 신청
```

### 유의사항

```text
대기 신청은 예약 확정이 아니에요.
빈자리가 생기면 제한 시간 동안 예약 기회를 드려요.
```

## C-12. 예약 대기 상세

### 목적

### 주요 정보

* 대기 상태
* 매장
* 서비스
* 희망 날짜
* 가능한 시간 범위
* 선호 직원
* 신청 시각
* 대기 순서 또는 안내 문구

대기 순서가 절대적인 예약 우선순위가 아닐 경우 숫자를 과도하게 강조하지 않는다.

```text
신청 순서와 희망 조건을 함께 확인해 빈자리를 안내해 드려요.
```

### 주요 행동

* 조건 수정
* 대기 취소
* 매장 상세 보기

## C-13. 빈자리 제안

### 목적

예약 대기 고객이 발생한 빈자리를 확인하고 제한 시간 안에 수락하도록 한다.

### 화면 구성

* 강조된 상태 문구
* 남은 수락 시간
* 매장
* 서비스
* 직원
* 날짜와 시간
* 예상 소요 시간
* 예약 정책

### 대표 문구

```text
원하던 시간에 빈자리가 생겼어요
```

### 타이머

```text
02:41 안에 선택해 주세요
```

타이머가 0이 되면 자동으로 만료 상태로 전환한다.

### 주요 버튼

```text
이 시간 예약하기
이번에는 건너뛰기
```

### 만료된 경우

```text
빈자리 제안 시간이 끝났어요.
대기 신청은 계속 유지돼요.
```

버튼:

```text
대기 상태 확인
```

### 수락 충돌

수락하는 사이 다른 상태 변경으로 예약이 불가능해진 경우:

```text
아쉽게도 이 자리를 더 이상 예약할 수 없어요.
대기 신청은 그대로 유지됩니다.
```

# 9. 운영자 화면 상세 기획

## A-02. 오늘의 운영 홈

### 목적

직원이 로그인 직후 오늘 필요한 업무를 파악하도록 한다.

### 상단 요약 카드

* 오늘 예약 수
* 현재 현장 대기
* 체크인 대기
* 진행 중 서비스
* 노쇼 후보
* 빈자리 제안 대기

### 시간순 운영 목록

예약과 현장 대기를 시간순으로 함께 표시한다.

카드 예시:

```text
14:00 예약
김자리 · 커트 · 민지 디자이너
체크인 대기
```

```text
현장 대기 12번
이자리 · 염색
앞선 팀 1
```

### 긴급 상태

화면 상단에 다음 상황을 경고한다.

* 호출 시간이 지난 고객
* 체크인하지 않은 예약
* 빈자리 제안 처리 실패
* 직원 일정 충돌
* 예약 처리 실패

## A-03. 예약 캘린더

### 목적

직원별 예약과 빈 시간을 한눈에 보고 일정을 관리하도록 한다.

### 보기 방식

* 일간 보기
* 주간 보기
* 직원별 열
* 시간축 행

### 예약 블록 정보

* 고객명
* 서비스
* 예약 상태
* 시작 및 종료 시간
* 체크인 여부
* 예약 경로

### 블록 상태

* 임시 예약
* 예약 확정
* 체크인
* 서비스 진행
* 완료
* 취소
* 노쇼

### 주요 행동

* 예약 상세 열기
* 빈 시간에 수동 예약 등록
* 직원별 보기 필터
* 상태별 필터
* 현재 시각으로 이동

### 일정 이동

드래그 앤 드롭은 초기 버전에서는 제공하지 않거나, 변경 확인 절차를 반드시 둔다.

예약 시간을 변경하면 다음을 확인한다.

```text
예약 시간을 변경할까요?

기존 시간: 오후 2:00
변경 시간: 오후 3:00

고객에게 변경 안내가 전송됩니다.
```

## A-04. 예약 상세

### 목적

직원이 예약 상태를 확인하고 현장 운영에 필요한 처리를 수행하도록 한다.

### 기본 정보

* 고객명
* 연락처
* 서비스
* 담당 직원
* 예약 시간
* 예약 경로
* 요청사항
* 예약 상태
* 체크인 상태

### 주요 행동

상태에 따라 다음 버튼을 제공한다.

* 체크인 처리
* 서비스 시작
* 서비스 완료
* 예약 시간 변경
* 담당 직원 변경
* 예약 취소
* 노쇼 처리
* 고객에게 연락

### 상태 변경 확인

노쇼 처리 예시:

```text
이 예약을 노쇼로 처리할까요?

고객이 매장에 도착하지 않았는지 다시 확인해 주세요.
```

### 상태 이력

하단에 주요 변경 기록을 시간순으로 표시한다.

```text
13:40 고객 체크인
13:35 직원이 예약 내용을 확인함
7월 10일 고객 예약 확정
```

## A-05. 수동 예약 등록

### 목적

전화나 현장 요청으로 들어온 예약을 직원이 대신 등록하도록 한다.

### 입력 항목

* 고객 검색 또는 신규 고객
* 서비스
* 담당 직원
* 날짜
* 시간
* 고객 요청사항
* 알림 발송 여부

### 충돌 표시

# 14. 반응형 화면 기준

## 고객용

모바일 우선으로 설계한다.

주요 행동 버튼은 화면 하단에 고정할 수 있다.

예약 시간 선택과 현장 호출 화면은 한 손으로 조작하기 쉬워야 한다.

## 운영자용

태블릿과 데스크톱을 우선으로 설계한다.

* 데스크톱: 일정과 상세 패널을 함께 표시
* 태블릿: 목록과 상세를 전환
* 모바일: 긴급 현장 처리 기능 위주로 제공

모바일 운영자 화면에서 우선 제공할 기능:

* 오늘 일정
* 고객 체크인
* 현장 호출
* 서비스 시작 및 완료
* 예약 상세 확인

복잡한 정책과 직원 일정 설정은 데스크톱 중심으로 제공한다.

# 15. 접근성 기준

* 상태를 색상으로만 구분하지 않는다.
* 버튼과 텍스트의 대비를 충분히 확보한다.
* 터치 영역은 최소 크기를 유지한다.
* 남은 시간은 숫자와 문장으로 함께 표시한다.
* 자동 갱신되는 대기 정보가 화면을 과도하게 흔들지 않도록 한다.
* 중요한 상태 변경은 스크린 리더가 인식할 수 있도록 안내한다.
* 시간 선택 버튼에 선택 가능 여부를 텍스트로 제공한다.

# 17. 핵심 프로토타입 시나리오

초기 디자인 프로토타입은 다음 다섯 가지 흐름을 우선 검증한다.

## 프로토타입 1. 예약 성공

```text
매장 선택
→ 서비스 선택
→ 시간 선택
→ 예약 확정
→ 예약 상세 확인
```

검증할 내용:

* 예약 과정이 너무 길지 않은가
* 선택한 정보가 각 단계에서 유지되는가
* 최종 확정 전 정보가 명확한가

## 프로토타입 2. 예약 시간 경쟁 실패

```text
시간 선택
→ 예약 확정
→ 다른 고객이 먼저 예약
→ 새로운 시간 선택
```

검증할 내용:

* 사용자가 실패 원인을 이해하는가
* 이전 화면으로 자연스럽게 돌아가는가
* 대기 신청으로 이어질 수 있는가

## 프로토타입 3. 빈자리 제안

```text
알림 수신
→ 제안 상세
→ 제한 시간 확인
→ 예약 수락
```

검증할 내용:

* 긴급성이 지나치게 불안하게 표현되지 않는가
* 남은 시간이 명확한가
* 대기 신청과 예약 확정의 차이를 이해하는가

## 프로토타입 4. 현장 대기와 호출

```text
QR 진입
→ 현장 대기 등록
→ 대기 상태 확인
→ 호출 수신
→ 입장 확인
```

검증할 내용:

* 대기 번호와 예상 시간이 잘 보이는가
* 호출 상태를 놓치지 않는가
* 응답 행동이 명확한가

## 프로토타입 5. 운영자 현장 처리

```text
운영 홈
→ 고객 체크인
→ 현장 고객 호출
→ 서비스 시작
→ 서비스 완료
```

검증할 내용:

* 현장에서 적은 조작으로 처리할 수 있는가
* 예약과 현장 대기를 혼동하지 않는가
* 상태 변경 실수를 예방할 수 있는가

# 20. 화면 설계의 핵심 기준

자리요의 디자인은 보기 좋은 예약 앱을 만드는 것보다 사용자가 현재 상황을 오해하지 않도록 만드는 것이 중요하다.

모든 화면은 다음 질문에 즉시 답할 수 있어야 한다.

* 지금 내 상태는 무엇인가?
* 다음에 무엇을 해야 하는가?
* 언제까지 행동해야 하는가?
* 행동하지 않으면 어떻게 되는가?
* 요청이 정상적으로 처리되었는가?
* 문제가 발생했다면 내 예약이나 대기 상태는 안전한가?

자리요의 핵심 사용자 경험은 예약 버튼을 누르는 순간이 아니라, 예약 경쟁이나 취소, 빈자리 제안, 호출 지연과 같은 예외 상황에서도 사용자가 자신의 상태를 신뢰할 수 있게 만드는 데 있다.
