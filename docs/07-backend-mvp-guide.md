# 자리요 백엔드 MVP 개발 가이드

## 1. 문서 목적

본 문서는 자리요 백엔드의 MVP를 가장 적은 범위로 안정적으로 출시하기 위한 개발 기준을 정리한다.

이 문서는 기능 목록보다 다음 질문에 답하는 데 집중한다.

* MVP에 반드시 포함할 기능은 무엇인가
* 무엇을 먼저 만들고 무엇을 뒤로 미룰 것인가
* 병렬 개발 전에 무엇을 합의해야 하는가
* MVP 완료 기준을 어디에 둘 것인가

---

## 2. MVP 원칙

* 핵심 흐름은 예약이다.
* 예약이 막히면 대기와 현장 운영으로 이어진다.
* 고도화된 자동화보다 수동 운영 가능성을 먼저 확보한다.
* 통계, 쿠폰, 결제, 반복 예약은 MVP에서 제외한다.
* 알림은 필수 이벤트만 지원하고 나머지는 후순위로 둔다.

---

## 3. MVP 실행 체크리스트

### 3.1 선행 합의

* 예약, 대기, 현장 대기 상태값은 문서의 기존 최소셋을 MVP 기본안으로 고정한다.
* 공통 오류 코드는 도메인별로 분리해 정의한다.
* 멱등성은 예약 생성, 취소, 대기 신청, 호출 같은 쓰기 API 전체에 적용한다.
* 예약 가능 시간 계산은 영업시간, 직원 근무시간, 기존 예약 충돌 중심의 1차 기준으로 시작한다.
* 고객과 운영자는 기본 분리하되, 한 사용자가 고객이면서 특정 매장의 직원일 수 있게 허용한다.

선행 합의 결과를 기준으로 후속 구현은 다음과 같이 진행한다.

* 상태값은 최소셋을 우선 적용하고, 추가 상태는 후속 논의로 미룬다.
* 오류 코드는 공통 분류보다 도메인별 세부 코드로 확장한다.
* 멱등성은 모든 쓰기 API에 일관되게 적용한다.
* 예약 가능 시간 계산은 단순 충돌 기준을 먼저 구현한다.
* 고객과 운영자 권한은 분리하되 사용자 계정은 겸용을 허용한다.

### 3.2 1차 구현

* 회원가입, 로그인, 내 정보 조회
* 매장 목록과 매장 상세 조회
* 서비스 목록과 직원 목록 조회
* 예약 가능 시간 조회
* 예약 생성, 조회, 취소

구현 메모:

* 2026-07-17 기준으로 매장 운영 도메인 조회용 스키마와 공통 조회 API 골격을 추가했다.

### 3.3 2차 구현

* 예약 대기 신청, 조회, 취소
* 빈자리 제안 수락
* 현장 대기 등록, 조회
* 고객 호출과 체크인

### 3.4 3차 구현

* 서비스 시작과 완료
* 운영자용 예약 및 대기 현황 조회

## 4. MVP 범위

### 포함 범위

MVP에는 아래 기능만 포함한다.

* 회원가입, 로그인, 내 정보 조회
* 매장 목록과 매장 상세 조회
* 서비스 목록과 직원 목록 조회
* 예약 가능 시간 조회
* 예약 생성
* 예약 조회
* 예약 취소
* 예약 대기 신청
* 예약 대기 조회 및 취소
* 현장 대기 등록
* 현장 대기 조회
* 고객 호출
* 체크인
* 서비스 시작
* 서비스 완료
* 운영자용 예약 및 대기 현황 조회

### 제외 범위

MVP에서 제외한다.

* 결제 및 정산
* 쿠폰과 포인트
* 복잡한 자동 배정
* 반복 예약
* 고도화된 통계 대시보드
* 다채널 알림 확장
* 세분화된 감사 리포트

## 5. MVP 핵심 흐름

### A. 고객 예약 흐름

1. 고객이 매장과 서비스를 조회한다.
2. 가능한 예약 시간을 확인한다.
3. 예약을 생성한다.
4. 예약 상세와 상태를 확인한다.
5. 필요 시 예약을 취소한다.

### B. 예약 대기 흐름

1. 원하는 시간이 없으면 예약 대기를 신청한다.
2. 빈자리가 생기면 대기 고객에게 기회를 제공한다.
3. 고객은 제안을 수락하거나 취소할 수 있다.

### C. 현장 운영 흐름

1. 현장 고객이 대기를 등록한다.
2. 운영자가 순서를 확인하고 호출한다.
3. 고객이 체크인한다.
4. 운영자가 서비스를 시작하고 완료 처리한다.

---

## 6. MVP 워크스트림

### A. 공통 기반 워크스트림

목표:

* 인증, 권한, 응답, 오류, 멱등성의 기준을 먼저 고정한다.

주요 범위:

* 로그인과 토큰 처리
* 공통 응답 구조
* 도메인별 공통 오류 코드 기준
* 권한 체크
* 매장 소속 검증
* `Idempotency-Key` 처리

완료 기준:

* 모든 MVP API가 같은 응답 규칙을 따른다.
* 중복 요청 방지 규칙이 정리된다.
* 운영자와 고객 권한 경계가 명확하다.

---

### B. 매장과 사용자 워크스트림

목표:

* 고객이 예약을 시작할 수 있는 최소 조회 데이터를 만든다.

주요 범위:

* 회원가입
* 로그인
* 내 정보 조회
* 매장 목록 조회
* 매장 상세 조회
* 서비스 목록 조회
* 직원 목록 조회

완료 기준:

* 고객이 로그인 후 예약 진입에 필요한 기본 정보를 조회할 수 있다.

---

### C. 예약 계산 워크스트림

목표:

* 예약 가능 시간 조회를 제공한다.

주요 범위:

* 영업시간
* 직원 근무시간
* 기존 예약 반영

핵심 합의 사항:

* 시간 슬롯 단위
* 시작 시간 정렬 규칙
* 직원 기준 계산 여부
* 서비스 기준 계산 여부

완료 기준:

* 동일 입력에 대해 동일 결과가 반환된다.
* 예약 생성 전에 검증 가능한 수준의 슬롯 정보가 나온다.

---

### D. 예약 워크스트림

목표:

* MVP의 핵심인 예약 생성, 조회, 취소를 안전하게 구현한다.

주요 범위:

* 예약 생성
* 예약 상세 조회
* 내 예약 목록 조회
* 예약 취소
* 예약 상태 이력

핵심 고려사항:

* 동시 예약 충돌 방지
* 중복 요청 방지
* 취소 가능 시간 검증
* 예약 상태 전이 통제

완료 기준:

* 같은 시간대 예약 충돌이 안전하게 처리된다.
* 고객이 자신의 예약을 안정적으로 조회하고 취소할 수 있다.

---

### E. 예약 대기 워크스트림

목표:

* 예약이 마감된 경우 대기 흐름을 제공한다.

주요 범위:

* 예약 대기 신청
* 예약 대기 조회
* 예약 대기 취소
* 빈자리 제안 생성
* 빈자리 제안 수락

핵심 고려사항:

* 제안 만료와 수락 충돌
* 한 빈자리에 대한 중복 확정 방지
* 예약 취소와의 연동

완료 기준:

* 예약 취소 후 대기 고객에게 기회를 줄 수 있다.
* 수락된 제안이 실제 예약으로 전환된다.

---

### F. 현장 운영 워크스트림

목표:

* 매장 직원이 현장 방문 고객을 처리할 수 있게 한다.

주요 범위:

* 현장 대기 등록
* 현장 대기 조회
* 고객 호출
* 체크인
* 서비스 시작
* 서비스 완료

핵심 고려사항:

* 호출과 체크인의 중복 처리 방지
* 예약 고객과 현장 고객 흐름 분리
* 상태 전이 이력 저장

완료 기준:

* 운영자가 현장 고객을 호출하고 처리 완료까지 진행할 수 있다.

MVP 구현 결정:

* 로그인 고객은 직접 등록하고 비회원은 직원이 대리 등록한다.
* 현장 고객의 수동 체크인, 보류·복귀, 재호출, 노쇼, 서비스 완료까지 이 워크스트림에 포함한다.
* 예약 고객 체크인과 서비스 진행은 예약 워크스트림에서 확장한다.
* 수동 대기 순서 변경, QR·키오스크 체크인, 실제 알림 발송은 후속 범위다.

---

### G. 운영자 조회 워크스트림

목표:

* 운영자가 당일 운영을 볼 수 있는 최소 화면 데이터를 제공한다.

주요 범위:

* 날짜별 예약 조회
* 날짜별 대기 조회
* 고객 호출 현황
* 체크인 현황

완료 기준:

* 운영자가 별도 집계 없이 당일 운영 상태를 확인할 수 있다.

---

## 7. 구현 순서

MVP는 아래 순서로 진행하는 것이 안전하다.

1. 공통 기반
2. 사용자와 매장 조회
3. 예약 가능 시간 계산
4. 예약 생성 및 조회
5. 예약 취소
6. 예약 대기
7. 현장 대기와 호출
8. 체크인 및 서비스 상태 변경
9. 운영자 조회

이 순서를 권장하는 이유는 예약 계산과 상태 전이 규칙이 뒤에서 다시 바뀌는 비용이 크기 때문이다.

## 8. 먼저 합의해야 하는 항목

MVP 구현 전에 아래 항목은 고정해야 한다.

* 예약 상태값
* 대기 상태값
* 현장 운영 상태값

* 멱등성 키 적용 범위

이 항목이 흔들리면 예약, 대기, 운영 흐름이 동시에 흔들린다.

---

## 9. MVP 완료 기준

MVP 완료는 다음 조건을 만족할 때로 본다.

* 고객이 로그인 후 매장을 조회하고 예약할 수 있다.
* 예약이 마감되면 대기 또는 현장 대기로 이어진다.
* 운영자가 예약과 현장 고객을 한 흐름으로 처리할 수 있다.
* 핵심 API가 공통 오류와 권한 규칙을 일관되게 사용한다.
* 중복 요청과 동시성 충돌이 치명적 데이터 오류로 이어지지 않는다.
