# 자리요 백엔드 개발 워크스트림 가이드

## 1. 문서 목적

본 문서는 자리요 백엔드 구현을 여러 명이 병렬로 진행할 때 참고하는 작업 가이드다.

이 문서는 순차 개발 계획보다 다음 항목에 집중한다.

* 어떤 작업 트랙으로 나눌지
* 각 트랙의 선행 조건이 무엇인지
* 병렬 진행이 가능한 범위가 어디까지인지
* 완료 기준을 무엇으로 잡을지

---

## 2. 공통 원칙

* CRUD보다 도메인 흐름을 우선한다.
* 화면 개수보다 상태 전이 안정성을 우선한다.
* 예약 가능 시간은 저장보다 계산을 우선한다.
* 동시성, 멱등성, 권한 검증은 초반부터 반영한다.
* 알림과 통계는 핵심 예약 흐름이 안정화된 뒤 확장한다.

---

## 3. 작업 방식

백엔드 구현은 `단계`보다 `워크스트림` 기준으로 나눈다.

각 워크스트림은 아래 항목으로 관리한다.

* 목표
* 주요 범위
* 선행 조건
* 병렬 진행 가능 여부
* 완료 기준
* 의존 관계

권장 방식:

* 공통 기반을 먼저 짧게 정리한다.
* 이후 예약, 대기, 현장 운영, 알림 영역은 담당자를 나눠 병렬 진행한다.
* 상태값, 이벤트명, 응답 구조처럼 다른 트랙에 영향을 주는 항목은 먼저 합의한다.

---

## 4. 워크스트림 정의

### A. 공통 기반 워크스트림

목표:

* 모든 도메인 구현의 공통 바닥을 만든다.

주요 범위:

* 인증 방식
* 공통 응답 포맷
* 공통 예외 처리
* 오류 코드 규칙
* 권한 체크 구조
* 멀티테넌트 검증
* `Idempotency-Key` 처리 구조
* 감사 로그 또는 상태 이력 공통 방식

선행 조건:

* 없음

병렬 진행:

* 가장 먼저 시작해야 한다.
* 다른 트랙은 이 워크스트림의 기본 방향이 정해진 뒤 진행하는 것이 안전하다.

완료 기준:

* 인증/인가 구조가 정리됨
* 공통 오류 응답 구조가 고정됨
* 멱등성 처리 방식이 합의됨
* 매장 소속 검증 방식이 고정됨

의존 관계:

* 대부분의 워크스트림이 이 트랙을 참조한다.

---

### B. 매장 운영 도메인 워크스트림

목표:

* 예약 계산과 운영 기능의 기반 데이터를 구현한다.

주요 범위:

* 매장
* 매장 정책
* 서비스
* 직원
* 직원 가능 서비스
* 영업시간
* 직원 스케줄
* 일정 예외

선행 조건:

* 공통 기반 워크스트림

병렬 진행:

* 사용자/권한 트랙과는 어느 정도 병렬 진행 가능
* 예약 계산 트랙과는 강한 의존 관계가 있음

완료 기준:

* 매장 상세, 서비스, 직원, 운영 정책 데이터를 안정적으로 조회 가능
* 예약 가능 시간 계산에 필요한 입력 데이터가 모두 준비됨

의존 관계:

* 예약 계산
* 예약
* 현장 운영

구현 메모:

* 2026-07-17 기준으로 매장, 매장 정책, 서비스, 직원 가능 서비스, 영업시간, 직원 스케줄, 일정 예외의 조회 기반 스키마와 API 골격을 추가했다.

---

### C. 사용자 및 권한 워크스트림

목표:

* 고객과 매장 멤버를 분리된 역할로 안전하게 다룬다.

주요 범위:

* 사용자 계정
* 고객 프로필
* 매장 멤버
* 역할별 접근 제어
* 로그인 사용자 기준 내 정보 조회

선행 조건:

* 공통 기반 워크스트림

병렬 진행:

* 매장 운영 도메인과 병렬 진행 가능
* 예약, 대기, 운영자 API 구현 전에는 방향이 정리되어야 함

완료 기준:

* 고객용 API와 운영자용 API 권한 분기 가능
* 한 사용자가 고객이면서 특정 매장의 직원일 수 있는 구조 확보

의존 관계:

* 예약
* 예약 대기
* 현장 운영

---

### D. 예약 계산 워크스트림

목표:

* 프론트엔드가 사용할 예약 가능 시간 계산 기능을 제공한다.

주요 범위:

* 날짜별 예약 가능 슬롯 계산
* 직원별 가능 시간 계산
* 서비스별 가능 시간 계산

필수 입력:

* 영업시간
* 직원 근무시간
* 일정 예외
* 서비스 소요 시간
* 기존 예약

선행 조건:

* 공통 기반 워크스트림
* 매장 운영 도메인 워크스트림

병렬 진행:

* 예약 도메인 트랙과 일부 병렬 진행 가능
* 단, 슬롯 검증 규칙은 예약 트랙과 반드시 같이 맞춰야 함

완료 기준:

* 같은 조건에서 일관된 슬롯 계산 결과를 반환함
* 프론트엔드가 예약 가능 시간 조회 API를 사용할 수 있음

의존 관계:

* 예약
* 예약 대기

---

### E. 예약 워크스트림

목표:

* 예약 생성부터 취소까지 핵심 예약 흐름을 안전하게 구현한다.

주요 범위:

* 예약 생성
* 예약 조회
* 예약 취소
* 예약 상태 변경
* 예약 상태 이력

MVP-P0 구현 기준:

* 일반 예약은 홀드 없이 즉시 `CONFIRMED`로 생성
* 직원과 고객 단위 동시성 제어 후 충돌 재검사
* 고객 취소는 자유 문장 사유만 저장
* 생성 이력은 `CREATED`, 고객 취소 이력은 `CUSTOMER_CANCELLED`
* 예약 홀드와 목록 커서 페이지네이션은 후속 작업으로 분리

핵심 고려사항:

* 중복 예약 방지
* 동시성 제어 전략
* 멱등성 키 재사용 검증
* 취소 가능 시간 정책 검증

선행 조건:

* 공통 기반 워크스트림
* 사용자 및 권한 워크스트림
* 예약 계산 워크스트림

병렬 진행:

* 예약 대기 트랙과 일부 병렬 가능
* 상태값과 이벤트는 먼저 합의해야 함

완료 기준:

* 같은 슬롯에 대한 동시 예약 요청을 안전하게 차단 가능
* 예약 생성, 조회, 취소 API 동작 가능
* 상태 이력이 남음

의존 관계:

* 예약 대기
* 알림
* 운영 기능

---

### F. 예약 대기 및 빈자리 제안 워크스트림

목표:

* 예약 취소와 빈자리 재배정을 연결한다.

주요 범위:

* 예약 대기 신청
* 예약 대기 취소
* 후보 탐색
* 빈자리 제안 생성
* 제안 수락
* 제안 만료 처리

핵심 고려사항:

* 제안 만료와 수락의 충돌
* 하나의 빈자리에 대한 중복 확정 방지
* 예약 취소 이벤트와의 연결

선행 조건:

* 예약 워크스트림
* 예약 계산 워크스트림

병렬 진행:

* 현장 운영 트랙과는 비교적 독립적
* 알림 트랙과는 인터페이스만 먼저 맞추면 병렬 가능

완료 기준:

* 예약 취소 후 대기 고객에게 빈자리 제안 가능
* 제안 수락 시 실제 예약으로 안전하게 전환됨

의존 관계:

* 알림

---

### G. 현장 운영 워크스트림

목표:

* 예약 고객과 현장 고객의 방문 흐름을 운영 화면에서 처리할 수 있게 만든다.

주요 범위:

* 현장 대기 등록
* 대기 순번 조회
* 고객 호출
* 체크인
* 서비스 시작
* 서비스 완료
* 노쇼 처리

핵심 고려사항:

* 호출 상태 중복 변경 방지
* 체크인과 서비스 상태 전이 규칙
* 예약 고객과 현장 고객 흐름의 분리

선행 조건:

* 공통 기반 워크스트림
* 사용자 및 권한 워크스트림
* 매장 운영 도메인 워크스트림

병렬 진행:

* 예약 대기 트랙과는 비교적 독립적으로 진행 가능

완료 기준:

* 운영자가 대기 등록, 호출, 체크인, 완료까지 처리 가능
* 상태 전이가 이력으로 남음

의존 관계:

* 알림
* 운영 기능

구현 경계:

* 이슈 #10은 현장 고객의 등록, 호출, 체크인, 서비스 시작·완료, 노쇼까지 완결한다.
* 예약 고객 체크인·노쇼·서비스 진행은 예약 모델을 소유하는 이슈 #9에서 구현한다.
* 비회원은 직원 대리 접수만 허용하고 공개 비회원 관리 토큰은 후속 범위로 둔다.
* 쓰기 API 멱등성은 PostgreSQL에 요청 해시와 성공 응답을 저장하는 방식으로 보장한다.

### H. 알림 및 비동기 처리 워크스트림

목표:

* 핵심 이벤트를 사용자 알림과 후속 처리로 연결한다.

주요 범위:

* 예약 확정 알림
* 예약 임박 알림
* 빈자리 제안 알림
* 현장 호출 알림
* 실패 작업 재처리

핵심 고려사항:

* 알림 실패가 도메인 트랜잭션 실패가 되지 않아야 함
* 이벤트 발행과 소비 경계 분리

선행 조건:

* 예약, 예약 대기, 현장 운영 중 최소 하나 이상의 이벤트 소스 구현

병렬 진행:

* 도메인 이벤트 명세가 정해지면 병렬 진행 가능

완료 기준:

* 핵심 이벤트 발생 후 알림 작업이 비동기로 연결됨
* 실패한 작업 재시도 가능

의존 관계:

* 예약
* 예약 대기
* 현장 운영

완료 기준:

## 5. MVP 범위 권장안

여러 명이 병렬 작업하더라도 MVP 범위는 먼저 고정하는 것이 좋다.

MVP 권장 포함:

* 매장/서비스/직원 조회
* 예약 가능 시간 조회
* 예약 생성 및 취소
* 내 예약 조회
* 예약 대기 신청
* 현장 대기 등록
* 운영자 예약/대기 조회
* 고객 호출
* 체크인

MVP 후순위:

* 고도화된 통계
* 복잡한 자동 배정
* 다채널 알림 확장
* 반복 예약
* 쿠폰, 결제, 정산
